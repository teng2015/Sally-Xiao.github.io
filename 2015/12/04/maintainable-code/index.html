第8章 避免“空比较”

什么是空比较

函数本身期待的是一个数组，空比较并不能避免出错，但有一点另外，如果数据本身期望的是null或者非null，比如节点；还是可以直接用===null或者!==null来处理。

有效比较

  • 1、对于字符串，数字，布尔值，undefined直接用typeof来判断。
  • 2、引用类型, 使用instanceof

必须使用Error对象，如下所示

何时应该抛出错误

  • 1、一旦修复了一个很难调试的错误，尝试增加一两个自定义错误。当再次发生错误时，这将有助于更容易地解决问题；
  • 2、如果增加编写代码，思考一下：“我希望[某些事情]不会发生，如果发生，我的代码会一团糟糕”。这时，如果“某些事情”发生，就抛出一个错误；
  • 3、如果正在编写的代码别人（不知道是谁）也会使用，思考一下他们使用的方式，在特定情况下抛出错误；

该使用try…catch还是throw?

比较深层次的调用就用throw把错误抛出，最上层的错误还是要用try..catch捕获住。

什么是门面模式

门面模式的意思是在一个对象上进行封装方法，然后把这些方法暴露给其他人用。为了避免直接操作原有对象上的方法。如下所示：

什么是Polyfill

针对一些现代浏览器实现了的方法在低版本的浏览器下模拟实现，达到兼容的目的；不过可能会实现的方法跟原生有出入，导致使用上出现问题。

阻止修改

  • 1、阻止扩展（ Object.preventExtension() 和 Object.isExtensible()）
  • 2、密封( Object.seal() )
  • 3、冻结( Object.freeze() )
    兼容：IE9+、Firefox 4+、Safari 5.1+、Opera 12+ 和 chrome

浏览器嗅探

User-Agent检测

  • 1、应该只检测IE8及以下的浏览器，不要试图去检测IE9及更高的版本；
  • 2、几乎所有的浏览器的User-Agent都可以被工具所更改；

特性检测

特性检测比User-Agent检测更好的原因是只有当某个特性存在我才使用某个特性，不存在则不使用，可以避免浏览器升级导致某些特性发生变化，而导致代码发生错误。

避免浏览器推断

什么是浏览器推断？
根据某个浏览器特性去推断浏览器的版本，这好比根据是否有会说话来判断是否是一个人，以部分属性来推断某个事物，从逻辑上也说不通。

总结

最好的办法是使用特性检测，退而求其次是User-Agent检测，避免使用特性推断和浏览器推断。

文章和目录结构

最佳实践

  • 1、一个文件只包含一个对象
  • 2、相关文档用目录分组
  • 3、保持第三方代码独立
  • 4、确定创建位置。网站应当是可以配置的，使用编译后的目录而非源码目录。不要把编译后的内容提交到源码工程里。
  • 5、保持测试代码的完整性。你的javascript测试代码应该一并提交到项目工程中的一个显眼的位置。让开发人员很容易注意到遗漏测试的情况。

基本结构

  • 1、build
    用来放置最终构建后的文件，理想情况下这个目录不应该提交；
  • 2、src
    用来放置所有的源文件，包括用来进行文件分组的子目录；
  • 3、test或者tests
    用来放置所有的测试文件。通常包含一些同源代码目录一一对应的子目录或文件

自动化测试

前端测试的话题由来已久，关于一些背景和难题的介绍请阅读《Javascript Web富应用开发》

  • 1、YUI Test Selenium引擎

  • 2、Yeti
    用javascript写测试并在node.js上运行。需要有一个html文件来自动执行你的测试。

  • 3、PhantomJS
    PhantomJS是一个命令行版本的Webkit,尽管并非完全相同，但是它的表现跟webkit(chrome和safari的渲染引擎)相似，并且可以不打开浏览器就能执行真正的浏览器测试。 官网地址：http://www.phantomjs.org。附带的脚本可以在 Jasmine和QUnit 上运行测试。 <编写可维护Javascript>第19章 page186

  • 4、JsTestDriver
    官网地址http://code.google.com/p/js-test-driver。基于已安装的浏览器运行测试。

组装“构建系统”

编制打包计划

  • 1、开发
    开发版本应该是由开发人员驱动的，构建过程应该尽可能地快，以免打扰开发人员的开发过程。不应当超过15秒

  • 2、集成
    集成版本是一个按照固定时间表自动构建的版本。集成版本负责寻找整个系统中的问题。构建过程可以花费稍长时间但是必须尽可能的详尽。

  • 3、发布
    一个按需执行的构建，仅仅在推送到生产环境之前执行。获得代码的最终版本以便部署。
